Wichtigste ADRs im Detail
4 Accepted
ADR-001 · Accepted
Modularer Monolith
Entscheid: Ein Deployment-Artefakt. Module mit expliziten Service-Interfaces — kein shared State, kein direkter DB-Zugriff über Grenzen.
Warum: 4 Devs, 360 h, 3 Monate — Microservice-Overhead (Service Discovery, Tracing, Pipelines) nicht gerechtfertigt. Stateless-Design hält Tür für späteren Ausbau offen.
ADR-002 · Accepted
Python / FastAPI + React 18
Entscheid: Backend Python 3.13 / FastAPI ASGI, Batch-Response JSON. Frontend React 18 / TypeScript 5. Kein SSE.
Warum: KI-Ökosystem (LiteLLM, pypdf, python-docx) in Python nativ. Batch-Response ermöglicht fail-closed Konfidenz-Pipeline — SSE hätte sie unmöglich gemacht.
ADR-008 · Accepted
Fail-Closed Konfidenz-Pipeline
Entscheid: Mehrstufige Defense-in-Depth. Jede Stufe kann unterdrücken — Antwort nur wenn alle passieren.
Warum: Halluzinationsrate = 0 % ist das Produktversprechen. Falsch unterdrückte Antwort ist akzeptabel — ausgelieferte Halluzination nicht. Schwellen in DB kalibrierbar.
ADR-010 · Accepted
API-First mit OpenAPI 3.0
Entscheid: openapi.yaml vor der Implementierung — verbindlicher Vertrag. FastAPI Auto-Spec bewusst deaktiviert.
Warum: Paralleles Arbeiten ab Tag 1. Spec ist auch Kontext für Claude Code — ohne formalen Vertrag fehlt KI die Grundlage für konsistente Typen und Fehlerformate.

API-First (ADR-010)

openapi.yaml ist der Vertrag — vor Implementierung. Frontend und Backend arbeiten parallel. Spec = Kontext für Claude Code.

Bewusst NICHT verwendet
  • Kein CQRS — zu komplex für MVP; Read = Write Model genügt
  • Kein Event-Driven — kein Kafka/RabbitMQ; pg_notify ist Job-Trigger, kein Event-Bus
  • Kein SSE-Streaming — ADR-008 fail-closed erfordert Batch-Response
  • Kein separater Vektor-Store — pgvector genügt für < 10 000 Chunks im MVP
Fazit

Das Muster passt zum Team, Budget und PoC-Scope. Kein Wunschdenken — ein Draft, der wirklich gebaut werden kann.

Single Point of Failure?
L4 · Frage 01

PostgreSQL als SPOF — kein Replica, kein Cluster. Bei DB-Ausfall vollständig unavailable.

Mitigation: täglicher pg_dump-Cron. Akzeptiertes Risiko für Proof of Concept (< 30 Nutzer, Business Hours).

LLM-Provider-Abhängigkeit — bei OpenAI-Ausfall keine Antworten.

Mitigation: LiteLLM-Abstraktion ermöglicht schnellen Wechsel auf Azure EU oder Ollama per Config. Circuit Breaker im API Server.

Single-Instance-Deployment — kein Load Balancer, kein Horizontal Scaling.

Bewusste MVP-Entscheidung. Stateless API-Design (ADR-001) erlaubt späteres Scaling ohne Architektur-Umbau.

Worker Single-Threaded — parallele Uploads können Queue stauen.

Bei < 30 Nutzern kein Problem. pgqueuer skaliert horizontal mit mehreren Worker-Instanzen (Post-MVP).

Passt das Pattern wirklich?
L4 · Frage 03
Modularer Monolith ✓

4 Container mit klaren Grenzen — das sieht nach Microservices aus, ist aber kein verteiltes System. Kommunikation direkt über TCP (DB) und HTTPS. Kein Service Mesh.

Modul-Grenzen durch Service-Interfaces im Code — nicht durch Netzwerk-Isolation.

API-First ✓

openapi.yaml geschrieben bevor eine Zeile Backend-Code entsteht. Frontend mockt gegen Spec. Echter API-First — nicht nachträgliche Dokumentation.

Was wir bewusst nicht sind
  • Kein CQRS — Read = Write Model; MVP-Scope genügt
  • Kein Event-Driven — pg_notify ist Job-Trigger, kein Event-Bus
  • Keine Hexagonal Architecture — aber ADR-001-Interfaces zeigen dahin
Fazit

Das Pattern passt zum Team und Budget. Die Architektur ist ehrlich — kein Wunschdenken.

Wie testet ihr das?
L4 · Frage 05
Unit Tests

RAG-Komponenten isolierbar testbar: Chunking, Embedding, Retrieval, Scoring. Modulare Pipeline (ADR-001) macht das möglich.

Eval-Dataset + CI-Gate (ADR-009)

20–30 Testfragen aus PoC-Dokumenten, manuell erstellt im Spike-Kick-off. LLM-as-Judge bewertet. CI blockiert bei Regression.

Was ist schwer testbar?
  • LLM-Outputs non-deterministisch — mehrere Runs, statistische Schwellen statt Exact-Match
  • Retrieval-Qualität zeigt erst mit echten PoC-Dokumenten valide Werte
  • Konfidenz-Schwellen sind Hypothesen bis PoC-Erfahrung vorliegt
E2E-Pflichtszenarien
  • Happy Path: Frage → Antwort + Quellenlink + Seitenangabe
  • «Weiss ich nicht»-Case: ausserhalb Korpus → keine Halluzination
  • Upload: Dokument → in ≤ 5 Min abfragbar

Was würdet ihr heute anders machen?
L4 · Frage 06

Backup-Strategie von Anfang an — war ein Peer-Review-Finding, hätte von Beginn im Architecture Draft stehen sollen. Kein Betrieb ohne funktionierenden pg_dump.

US-06 (Stale-Content) direkt als Post-MVP — der SMTP-Provider-Entscheid blockierte Sprint 0 unnötig. Klare Post-MVP-Abgrenzung spart Scope-Diskussionen.

Team-Grösse von Anfang an korrekt — ADRs sagten «3 Devs», das Team hat 4 Personen. Falsche Constraints erzeugen falsche Entscheide.

ADR-Status-Disziplin — ADR-001 stand als «Proposed» obwohl im Architecture Draft bereits «Accepted». ADR-Status ist kein Formalismus — er steuert, was als bindend gilt.

Batch vs. SSE früher entscheiden — der Konflikt zwischen Streaming und Fail-Closed-Pipeline (ADR-002 vs. ADR-008) war lange offen. Batch wäre von Anfang an die richtige Wahl gewesen.
