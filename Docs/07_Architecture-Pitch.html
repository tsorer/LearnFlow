ADR-001 · Architekturstil

Modularer Monolith — 4 Container

webapp

React 18 / TS · Nginx · SSE-Consumer

api

Python 3.13 / FastAPI · RAG + Auth

worker

pgqueuer · async Processing

db

PostgreSQL 17 + pgvector

Klare Modulgrenzen über Service-Interfaces (kein DB-Zugriff über Grenzen) halten RAG-Komponenten austauschbar; ein Deployment-Artefakt macht Debugging und Onboarding einfach (docker compose up); das stateless Design hält den späteren Ausbau offen.

ADR-002 · Backend & Frontend

Warum Python im Backend?

  • De-facto-Standard für RAG/LLM — LangChain, LiteLLM, pypdf, python-docx, sentence-transformers sind direkt verfügbar, kein Adapter-Layer
  • FastAPI ist async-native (ASGI) — SSE-Streaming token-by-token ohne Zusatz-Libraries
  • Ein Backend in einer anderen Sprache müsste dieses Ökosystem über eine Brücke nachbauen — verbranntes Budget

Frontend: React 18 + TypeScript

Da der Code mit Claude Code entsteht, zählt die Trainingsdaten-Dichte (zuverlässige Generierung) und TypeScript als Review-Netz beim Prüfen des KI-Outputs. Vue / Svelte / HTMX wurden abgewogen.

ADR-003 · Datenpersistenz

PostgreSQL 17 — eine Datenbank für alles

Relational · users, documents, feedback, config Vektor · pgvector HNSW Volltext · tsvector/GIN (deutsch) Dateien · bytea

Ein Service → ein Backup, eine Verbindung, SQL-Joins über Vektor und Relation. Dedizierter Vector Store (Qdrant) bringt bei < 10 000 Chunks keinen Vorteil, nur Ops-Last. Schwellenwerte leben in der config-Tabelle — kalibrierbar ohne Deployment.

ADR-007 + ADR-008 · Retrieval & Antwortprüfung

Mehrstufige Prüfung vor jeder Antwort

0
Retrieval-Gate — erreicht kein Chunk die Similarity-Schwelle, wird ohne LLM-Aufruf „Weiss ich nicht" zurückgegeben
1
Retrieval-Konfidenz — Score aus Trefferzahl & Ähnlichkeit; unter Schwelle: keine Antwort
2
Citation-Check — prüft, ob jede Aussage durch zitierte Chunks gedeckt ist (Coverage ≥ 50 %)
3
LLM-Self-Check — zusätzlicher Verifikationsaufruf, nur bei Antworten nahe der Schwelle

Davor: Hybrid-Retrieval kombiniert semantische Suche (pgvector) mit deutscher Volltextsuche — letztere findet exakte Fachbegriffe und Akronyme, die die Vektorsuche verfehlt. Stufen 0–2 sind deterministisch, also reproduzierbar und testbar.

ADR-004 + ADR-005 · LLM & Embeddings

EU-first & provider-portabel

  • Azure OpenAI EU als Default — interne Dokumente bleiben in der EU (DSGVO)
  • LiteLLM als Abstraktion — Wechsel Azure ↔ OpenAI ↔ Ollama ist ein config-Eintrag
  • Ollama als OnPrem-Fallback, ganz ohne Cloud

Der wunde Punkt

Beim Indexieren geht der gesamte Korpus an den Provider — die datenintensivste Exposition. Genau deshalb ist der EU-Endpunkt hier kein „nice to have", sondern der Default.

Embedding: text-embedding-3-small (1536)
ADR-006 · Background-Worker

Dokument-Verarbeitung im Hintergrund

Der Upload-Request darf nicht blockieren — Parsing, Chunking und Embedding laufen asynchron in einem Worker. Die Job-Verwaltung übernimmt pgqueuer: eine Python-Bibliothek, die PostgreSQL selbst als Warteschlange nutzt (über LISTEN/NOTIFY und SKIP LOCKED) — statt eines separaten Message-Brokers wie Redis.

Kein zusätzlicher Service

Jobs sind PostgreSQL-Zeilen — kein Redis, kein Celery-Container

Persistenz inklusive

Jobs überleben Neustarts; ein Backup deckt alles

SLA erfüllt

Dokument in < 5 Min verfügbar (US-04)

ADR-009 · Verifikation

Die 0-%-Vorgabe messbar machen

  • Gold-Dataset — vom Fachbereich erstellte Testfragen mit bekannter richtiger Antwort/Quelle, plus Fragen, die der Korpus nicht beantworten kann
  • Automatische Messung — Werkzeug (RAGAS) prüft je Antwort, ob sie durch die Quellen gedeckt ist; dazu die „Weiss-ich-nicht"-Quote
  • CI-Gate — bei jeder Code-Änderung läuft dieser Test automatisch; tritt eine Halluzination auf, schlägt der Build fehl

Grenze der Aussage

Ein endliches Testset kann „0 %" nur zeigen, nicht beweisen. Deshalb zusätzlich: laufende Beobachtung im Betrieb über Nutzer-Feedback (US-03).

Ehrlichkeit zum Schluss

Was wir noch klären müssen

Vor dem Code

Gold-Dataset (Fachbereich), Datenmodell + OpenAPI-Kontrakt, bytea vs. lo

Im Spike (Woche 1)

RAG-Parameter kalibrieren, Embedding-Modell, Streaming ↔ Grounding-Check auflösen

Vor MVP

Auth-Rollenmodell, Rate-Limiting, Quiz-Generierung, Secrets, LangChain-Frage

Spannung: Token-Streaming (Performance) vs. Antwort erst prüfen, dann ausliefern (fail-closed) — früh zu entscheiden.

Zusammengefasst

Worauf die Architektur ausgelegt ist

  • Einfach im Betrieb — modularer Monolith, eine Datenbank, vier Container
  • Datenschutzkonform — EU-Datenresidenz, Provider per Konfiguration austauschbar
  • Überprüfbare Antwortqualität — mehrstufige Prüfung plus messbares CI-Gate
  • Wartbar & umkehrbar — Provider und Schwellenwerte ohne Code-Deployment änderbar
