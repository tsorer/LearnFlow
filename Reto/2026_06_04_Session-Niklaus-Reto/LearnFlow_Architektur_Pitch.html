Wichtigste ADRs im Detail

ADR-001 · Accepted

Modularer Monolith

Entscheid: Ein Deployment-Artefakt. Module mit expliziten Service-Interfaces — kein shared State, kein direkter DB-Zugriff über Modulgrenzen.

Warum: 4 Devs, 360 h Budget, 3 Monate — Microservice-Overhead (Service Discovery, verteiltes Tracing, separate Pipelines) ist schlicht nicht gerechtfertigt. Stateless-Design hält die Tür für spätere Extraktion offen.

ADR-002 · Accepted

Python / FastAPI + React 18

Entscheid: Backend Python 3.13/FastAPI (ASGI), Batch-Response JSON. Frontend React 18/TypeScript 5. Kein SSE-Streaming.

Warum: Das gesamte KI-Ökosystem (LiteLLM, pypdf, python-docx) ist in Python native verfügbar. Batch-Response ermöglicht die fail-closed Konfidenz-Pipeline (ADR-008) — SSE hätte sie unmöglich gemacht.

ADR-008 · Accepted

Fail-Closed Konfidenz-Pipeline

Entscheid: Mehrstufige Defense-in-Depth im API Server. Jede Stufe kann unterdrücken. Antwort nur wenn alle Stufen passieren.

Warum: Halluzinationsrate = 0% ist das zentrale Produktversprechen. Eine falsch unterdrückte Antwort ist akzeptabel — eine ausgelieferte Halluzination nicht. Schwellen in DB, kalibrierbar ohne Deployment.

ADR-010 · Accepted

API-First mit OpenAPI 3.0

Entscheid: openapi.yaml wird vor der Implementierung geschrieben — verbindlicher Schnittstellenvertrag. FastAPI Auto-Spec bewusst deaktiviert.

Warum: Paralleles Arbeiten von Frontend und Backend von Tag 1. Der Spec ist auch Kontext für Claude Code — ohne formalen Vertrag fehlt der KI die Grundlage für konsistente Typen und Fehlerformate.

  • Kein Event-Driven — kein Kafka/RabbitMQ im Stack
  • Kein SSE-Streaming — ADR-008 (fail-closed) erfordert Batch
  • Kein separater Vektor-Store — pgvector genügt für < 10'000 Chunks

Single Point of Failure?

PostgreSQL als SPOF — kein Replica, kein Read-Replica, kein Cluster. Bei DB-Ausfall ist LearnFlow vollständig unavailable.

Mitigation: täglicher pg_dump-Cron in Docker Compose. Akzeptiertes Risiko für Pilot (< 30 Nutzer, Business Hours).

LLM-Provider-Abhängigkeit — bei OpenAI-Ausfall keine Antworten möglich.

Mitigation: LiteLLM-Abstraktion (ADR-004) ermöglicht schnellen Wechsel auf Azure EU oder Ollama per Config-Änderung. Circuit Breaker im API Server.

Single-Instance-Deployment — kein Load Balancer, kein Horizontal Scaling.

Bewusste Entscheidung für MVP. Stateless API-Design (ADR-001) ermöglicht späteres Scaling ohne Architektur-Umbau.

Background Worker — Single-Threaded — parallele Uploads können Queue stauen.

SLA: ≤ 5 Min für 50 Seiten. Bei < 30 Nutzern und gelegentlichen Uploads kein Problem. pgqueuer skaliert horizontal mit mehreren Worker-Instanzen (Post-MVP).

Passt das Pattern wirklich?

Modularer Monolith ✓

4 Container mit klaren Grenzen (Web App, API, Worker, DB) — das sieht nach Microservices aus, ist aber kein verteiltes System. Alles kommuniziert direkt über TCP 5432 (DB) oder HTTPS. Kein Service Mesh, kein Netzwerk-Mesh.

Modul-Grenzen sind durch Service-Interfaces im Code durchgesetzt — nicht durch Netzwerk-Isolation.

API-First ✓

openapi.yaml wird geschrieben, bevor eine Zeile Backend-Code entsteht. Frontend mockt gegen die Spec. Das ist echter API-First — nicht «wir dokumentieren die fertige API».

Was wir bewusst nicht sind

  • Kein CQRS — Read-Model = Write-Model; für MVP ausreichend
  • Kein Event-Driven — pg_notify ist kein Event-Bus, nur ein Job-Trigger
  • Kein Hexagonal Architecture — aber ADR-001-Interfaces sind ein Schritt dahin

Fazit

Das C2-Diagram zeigt klare Container-Grenzen. Das Pattern passt zum Team, Budget und Pilotscope. Die Architektur ist ehrlich — kein Wunschdenken.

Wie testet ihr das?

Unit Tests — RAG-Komponenten

Chunking, Embedding, Retrieval, Scoring — jede Komponente isoliert testbar. Modulare RAG-Pipeline (ADR-001) macht das möglich: kein monolithischer God-Class-Code.

Eval-Dataset + CI-Gate (ADR-009)

20–30 Testfragen aus Pilot-Dokumenten, manuell erstellt im Spike-Kick-off. LLM-as-Judge bewertet Antworten. CI-Pipeline blockiert bei Regression — kein Merge ohne grünes Eval.

Was ist schwer testbar?

  • LLM-Outputs sind non-deterministisch — mehrere Runs, statistische Schwellen statt Exact-Match
  • Retrieval-Qualität hängt von Chunk-Qualität ab — Eval zeigt erst nach echten Pilot-Dokumenten valide Werte
  • Konfidenz-Kalibrierung — Schwellenwerte sind Hypothesen bis Pilot-Erfahrung vorliegt

E2E-Pflichtszenarien

  • Happy Path: Frage → Antwort + Quellenlink + Seitenangabe
  • «Weiss ich nicht»-Case: Frage ausserhalb Korpus → keine Halluzination
  • Upload-Flow: Dokument hochladen → in ≤ 5 Min abfragbar

Was würdet ihr heute anders machen?

Backup-Strategie von Anfang an — war ein Peer-Review-Finding, hätte von Beginn weg im Architecture Draft stehen sollen. Kein Betrieb ohne funktionierenden pg_dump.

US-06 (Stale-Content) direkt als Post-MVP — der SMTP-Provider-Entscheid blockierte Sprint 0 unnötig. Klare Post-MVP-Abgrenzung spart Scope-Diskussionen.

Team-Grösse von Anfang an korrekt — ADRs sagten «3 Devs», das Team hat 4 Personen (Frank, Niklaus, Reto, Christoph). Falsche Constraints in ADR-Kontext erzeugen falsche Entscheide.

ADR-Status-Disziplin — ADR-001 stand monatelang als «Proposed» in der Datei, obwohl er im Architecture Draft bereits «Accepted» war. ADR-Status ist kein Formalismus — er steuert, was als bindend gilt.

SSE vs. Batch-Response früher entscheiden — der Konflikt zwischen Streaming und Fail-Closed-Pipeline (ADR-002 vs. ADR-008) war lange offen. Batch-Response wäre von Anfang an die richtige Wahl gewesen.
