## Container-Analyse — 4 Fragen je Container

---

### 1 · Web App — React 18 / TypeScript 5 · Vite · Nginx

| Frage                                                              | Antwort                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Warum diese Technologie und nicht die bekannteste Alternative?** | React ist de-facto-Standard für SPAs mit SSE-Unterstützung. Die bekannteste Alternative wäre **Vue 3** — technisch gleichwertig, aber mit kleinerem Ökosystem. HTMX würde SSE-Streaming, Quiz-UI und Quellenhervorhebung mühsam machen: zu wenig reaktiv für diese Interaktionstiefe. Nginx als Static Server ist die einfachste, zuverlässigste Option gegenüber einem Node-Server.                                                                                                                                    |
| **Passt sie zu unseren Top-3 QAs?**                                | **Reliability:** React selbst trägt wenig zur Halluzinationsrate bei — das ist Backend-Sache. Das SSE-Streaming (`EventSource`) zeigt Antworten Token-by-Token, was Fehler sofort sichtbar macht. **Security:** Kein State auf dem Client der schützenswert ist; JWT im HTTP-Only-Cookie möglich. Aber: XSS-Risiken durch React sind real, wenn Quellentext ungeprüft als HTML gerendert wird. **Maintainability:** TypeScript erhöht Wartbarkeit; Vite macht lokale Dev-Loops schnell. Nginx ist trivial zu betreiben. |
| **Kann unser Team (3 Devs, 3 Monate) das wirklich bauen?**         | Ja — sofern React-Erfahrung vorhanden ist. Wenn das Team primär Backend ist, kostet die SSE-Integration (`EventSource` + Streaming-UI) und die Quellenhervorhebung (Highlighting im Dokument) 2–3 Wochen mehr als geplant. Das ist ein reales Risiko.                                                                                                                                                                                                                                                                   |
| **Grösstes Risiko**                                                | **Quellenhervorhebung im Dokument** ist komplex: PDF im Browser darstellen und Abschnitte visuell markieren ist nicht trivial (PDF.js, Viewport-Synchronisation). Wenn das unterschätzt wird, wird US-01 («Klick auf Quelle hebt Abschnitt hervor») zum MVP-Blocker.                                                                                                                                                                                                                                                    |

### 2 · API Server — Python 3.12 / FastAPI (ASGI)

| Frage                                                              | Antwort                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Warum diese Technologie und nicht die bekannteste Alternative?** | FastAPI ist die beste Python-Wahl für async SSE-Streaming + RAG-Pipeline. Die bekannteste Alternative wäre **Django REST Framework** — synchron-first, schwerer für SSE, deutlich mehr Boilerplate. Flask wäre leichter, aber ohne native async-Unterstützung und Typ-Annotationen. FastAPI gibt uns ASGI, Pydantic-Validierung und das gesamte Python-KI-Ökosystem (LiteLLM, pypdf, python-docx) ohne Adapter-Layer.                                                                                                                 |
| **Passt sie zu unseren Top-3 QAs?**                                | **Reliability:** FastAPI ermöglicht den mehrschichtigen Unterdrückungsmechanismus als klare Pipeline (Quellenprüfung → Konfidenz → Self-Check). Der Circuit Breaker für LiteLLM ist mit `asyncio`-Timeouts elegant umsetzbar. **Security:** RBAC-Middleware als Dependency Injection ist in FastAPI idiomatisch; JWT-Validierung per `python-jose` etabliert. **Maintainability:** LiteLLM-Abstraktion liegt im API Server — Provider-Wechsel per Konfig, kein Code-Change. Dependency Injection macht Mock-Objekte in Tests trivial. |
| **Kann unser Team (3 Devs, 3 Monate) das wirklich bauen?**         | Ja — Python ist für das Team am ehesten bekannt. FastAPI hat eine flache Lernkurve. Das Risiko liegt nicht in FastAPI selbst, sondern in der RAG-Pipeline: Chunking-Strategie, Konfidenz-Scoring und Self-Check sind keine Framework-Fragen — das sind algorithmische Entscheidungen die Zeit kosten.                                                                                                                                                                                                                                 |
| **Grösstes Risiko**                                                | **Konfidenz-Scoring ist undefiniert** (Risiko 2 aus Requirements). FastAPI kann nicht kompensieren, was konzeptionell noch offen ist. Wenn der Self-Check-Mechanismus (LLM-Selbstevaluation vs. semantische Ähnlichkeit) nicht vor Sprint 1 definiert ist, wird der API Server zum Bottleneck des gesamten Projekts — unabhängig vom Framework.                                                                                                                                                                                       |

### 3 · Background Worker — pgqueuer (PostgreSQL-nativer Job-Queue)

| Frage                                                              | Antwort                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Warum diese Technologie und nicht die bekannteste Alternative?** | Die bekannteste Alternative wäre **Celery + Redis**. pgqueuer verzichtet bewusst auf Redis als separaten Service: Jobs werden in einer PostgreSQL-Tabelle persistiert, `pg_notify` triggert sofortige Benachrichtigung. Das eliminiert eine externe Abhängigkeit (Redis), ein zweites Backup-Target und eine weitere Konfigurationsdimension — relevant bei Single-Instance MVP mit < 30 Nutzern. Celery wäre Overkill und bringt Komplexität, die bei dieser Last nie gebraucht wird. |
| **Passt sie zu unseren Top-3 QAs?**                                | **Reliability:** Jobs sind PostgreSQL-transaktional — kein Job geht verloren wenn der Worker crasht; beim Neustart läuft er einfach weiter. Das ist zuverlässiger als Redis-Queue-Verlust bei ungraceful Shutdown. **Security:** Kein separater Service bedeutet keine zusätzliche Angriffsfläche. **Maintainability:** Ein Backup, eine DB-Verbindungskonfiguration, kein zweiter Service. Weniger zu betreiben = weniger operative Last = mehr Zeit für Kernentwicklung.             |
| **Kann unser Team (3 Devs, 3 Monate) das wirklich bauen?**         | Ja — pgqueuer ist eine einfache Bibliothek, nicht ein neues Paradigma. Die Komplexität liegt im Worker-Inhalt: PDF-Parsing, Chunking-Strategie, Embedding-Generierung. Das ist fachliche Komplexität, keine Infrastruktur-Komplexität. Die Chunking-Strategie («offen» im C2-Dokument) muss vor Implementierungsbeginn entschieden werden.                                                                                                                                             |
| **Grösstes Risiko**                                                | **pgqueuer ist wenig verbreitet** — Stack Overflow-Treffer sind dünn, Community-Support begrenzt. Bei einem unerwarteten Bug in der Bibliothek ist das Team auf sich gestellt. Celery hat hier einen klaren Vorteil. Für MVP mit < 30 Nutzern ist das akzeptabel; bei skaliertem Betrieb müsste die Wahl überdacht werden.                                                                                                                                                             |

### 4 · Datenbank — PostgreSQL 16 + pgvector

| Frage                                                              | Antwort                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Warum diese Technologie und nicht die bekannteste Alternative?** | Die bekannteste Alternative für Vektor-Storage wäre **Pinecone** (dedizierter Vektordatenspeicher) oder **Weaviate**. pgvector hält relationale Daten, Vektoren und Original-Dokumente (bytea) in einer einzigen PostgreSQL-Instanz: ein Backup, eine Verbindungskonfiguration, kein zweiter Service. Pinecone wäre ein zusätzlicher Cloud-Service mit DSGVO-Implikationen und Latenzen zwischen DB und Vektordatenspeicher. Für < 10 000 Chunks und < 30 Nutzer ist pgvector mit HNSW-Index mehr als ausreichend.                                                                                                       |
| **Passt sie zu unseren Top-3 QAs?**                                | **Reliability:** HNSW-Index liefert Sub-100-ms-Latenz für Similarity Search — der 10-Sekunden-p95 ist erreichbar. Die `config`-Tabelle für Schwellenwerte ist zentral für die Reliability-Architektur (kalibrierbar ohne Deployment). **Security:** PostgreSQL RBAC ist erprobt; DSGVO-Compliance durch EU-Datenhaltung on-premise. Pseudonymisierung direkt in DB-Queries implementierbar. **Maintainability:** Alle Schwellenwerte in der `config`-Tabelle — Konfidenz, Stale-Schwellenwert, Cluster-Mindestgrösse — sofort wirksam ohne Neustart. Ein einziger Service statt zwei reduziert operative Last erheblich. |
| **Kann unser Team (3 Devs, 3 Monate) das wirklich bauen?**         | PostgreSQL kennt jeder im Team — das ist der stärkste Vorteil dieser Wahl. pgvector erfordert minimales Setup: Extension aktivieren, HNSW-Index anlegen, fertig. Das Schema ist überschaubar: users, documents, embeddings, feedback, config, quiz_questions. Das ist in Sprint 0/1 umsetzbar.                                                                                                                                                                                                                                                                                                                           |
| **Grösstes Risiko**                                                | **bytea für Original-Dokumente skaliert nicht.** 10 MB Dokumente als bytea in PostgreSQL zu speichern ist für < 30 Nutzer und einen Pilot-Bereich akzeptabel, aber ineffizient: Backup-Grösse wächst mit jedem Upload, DB-Dumps dauern länger. Post-MVP mit mehreren Bereichen wäre ein Object Storage (S3/MinIO) die richtigere Wahl. Dieser Umbau ist nicht trivial — der MVP legt hier eine technische Schuld an.                                                                                                                                                                                                     |

## Übersichtstabelle

| Container             | Technologie              | Begründung (Kern)                                               | Grösstes Risiko                                                   |
| --------------------- | ------------------------ | --------------------------------------------------------------- | ----------------------------------------------------------------- |
| **Web App**           | React 18 / TS 5 · Nginx  | SSE-native, interaktiv genug für Quiz + Quellenhervorhebung     | Quellenhervorhebung in PDFs unterschätzt → MVP-Blocker            |
| **API Server**        | Python 3.12 / FastAPI    | Async-native, Python-KI-Ökosystem direkt verfügbar, RBAC als DI | Konfidenz-Scoring noch undefiniert — kein Framework löst das      |
| **Background Worker** | pgqueuer                 | Kein Redis, transaktionale Jobs, ein Service weniger            | Kleine Community — bei Library-Bug ist das Team auf sich gestellt |
| **Datenbank**         | PostgreSQL 16 + pgvector | Alles in einem: relationale Daten + Vektoren + Dokumente        | bytea für Dokumente = technische Schuld; Post-MVP Umbau nötig     |
